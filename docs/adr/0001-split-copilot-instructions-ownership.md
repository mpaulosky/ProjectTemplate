# Split Copilot instructions: Squad-owned coordination vs. project-owned coding standards

`.github/copilot-instructions.md` was carrying both Squad's Coding-Agent protocol (canary check, team-context reads, capability self-check, PR/branch conventions) and this project's own .NET coding standards (tech stack, testing, guardrails) in one file. We discovered that `squad upgrade` performs a raw, unconditional overwrite of `.github/copilot-instructions.md` whenever `@copilot` is on the roster (`.squad/team.md` contains `🤖 Coding Agent`, which it already does here) — confirmed by reading the installed `@bradygaster/squad-cli@0.11.0` source (`upgrade.js`), not just its docs. There is no merge, diff, or marker support; a future upgrade would have silently deleted the project's custom rules.

We decided to move the project-specific rules into a new sibling file, `.github/instructions/dotnet-project.instructions.md` (`applyTo: "**"`), which Copilot still auto-loads for every request but which sits outside squad-cli's `TEMPLATE_MANIFEST` and is therefore immune to `squad upgrade`. `.github/copilot-instructions.md` now contains only Squad's Coding-Agent content, matching what `squad upgrade` would install anyway — making future upgrades a no-op instead of a silent data-loss event. The redundant, unwired duplicate `.github/squad-copilot-instructions.md` (which nothing auto-loaded) was deleted.

**Considered and rejected:** keeping everything in one file and instructing the user to never run `squad upgrade` with `@copilot` enabled — rejected because it fights the tool instead of working with its ownership model, and is easy to forget.
